# Container-Relative Scaling Migration

## Blueprint (Design)

### Context

Issues #28 and #29 changed the Lys CSS engine to use container-relative units (`cqi`) for font-size and padding defaults. Two categories of existing artifacts need updating to reflect this:

1. **Example decks** (#30) — All four themed examples (`themed.html`, `full.html`, `square.html`, `demo.html`) use hand-tuned `clamp()` values with `vw` units for heading sizes and spacing. These should be migrated to `em`-based sizing that scales with the new `--lys-font-size-base` token, so they work at any viewport and aspect ratio without manual tuning.

2. **Documentation and LLM deliverables** (#31) — The `lys-core.spec.md` Gherkin scenarios, `llms.txt` token table, and `skill/SKILL.md` token reference table still contain old default values (`2rem`, `clamp(1rem, 2vw, 1.5rem)`). These must be updated to match the actual CSS (`4cqi`, `clamp(0.75rem, 2.5cqi, 1.5rem)`).

Neither issue involves behavioral code changes — this is purely migration of existing content.

### Architecture

#### Example migration strategy

The key insight: with container-relative `--lys-font-size-base`, heading sizes should use `em` units relative to the base font size rather than independent `clamp()`/`vw` values. This way, all text scales together as the slide shrinks.

| Before (vw-based) | After (em-based) |
|---|---|
| `h1 { font-size: clamp(2rem, 5vw, 4rem); }` | `h1 { font-size: 2.5em; }` |
| `h2 { font-size: clamp(1.5rem, 3.5vw, 2.5rem); }` | `h2 { font-size: 2em; }` |
| `h3 { font-size: clamp(1.1rem, 2.2vw, 1.5rem); }` | `h3 { font-size: 1.5em; }` |
| `p { font-size: clamp(0.9rem, 1.8vw, 1.25rem); }` | `p { font-size: 1em; }` (inherits base) |
| `small { font-size: clamp(0.75rem, 1.2vw, 0.9rem); }` | `small { font-size: 0.85em; }` |

The exact `em` multipliers should be tuned per example to preserve the visual hierarchy. The values above are starting points.

For examples that override `--lys-font-size-base` (e.g., `square.html` with `1.125rem`), either:
- Remove the override and rely on the new `cqi` default, or
- Change it to a `cqi`-based value if the example needs a different scale.

#### Module boundaries

- `examples/*.html` — Theme CSS within `<style>` tags. No changes to markup structure.
- `specs/lys-core.spec.md` — Gherkin scenario text only.
- `llms.txt` — Token reference table (manually authored section).
- `skill/SKILL.md` — Token reference table (manually authored section).
- **No source code changes.** `src/lys.css` and `src/*.ts` are untouched.

### Anti-Patterns

- **Replacing `vw` with `cqi` in example heading styles.** The whole point is to use `em` units that scale relative to the base font size. Using `cqi` directly in heading styles couples them to the container rather than the typographic scale.
- **Removing all `clamp()` from examples.** Some uses (e.g., for spacing or layout gaps) may still benefit from `clamp()` with `cqi` or `em` bounds. Only text sizing should move to `em`.
- **Adding new `--lys-*` tokens.** Examples should use the existing token API. No new public API.
- **Changing example markup structure.** Only `<style>` content changes. The HTML structure is stable.
- **Hand-tuning for a specific viewport.** The goal is viewport-agnostic examples. If an `em` value only looks good at 1920px, it's wrong.

## Contract (Quality)

### Definition of Done

1. No example file contains `vw` units in text `font-size` declarations.
2. All example heading sizes use `em` units relative to the base font size.
3. Examples that override `--lys-font-size-base` either remove the override or use `cqi` units.
4. All five examples render without overflow at 1920x1080 (16:9).
5. All five examples render without overflow at 375x667 (mobile portrait).
6. `specs/lys-core.spec.md` Gherkin scenarios reference the correct token defaults.
7. `llms.txt` token table shows `4cqi` and `clamp(0.75rem, 2.5cqi, 1.5rem)`.
8. `skill/SKILL.md` token table shows `4cqi` and `clamp(0.75rem, 2.5cqi, 1.5rem)`.
9. `pnpm build` succeeds (site assembly includes updated examples).
10. No regressions in unit tests (token string matching tests must still pass).

### Regression Guardrails

- `examples/minimal.html` must remain a zero-config 4-line deck with no custom CSS.
- All examples must remain valid, self-contained HTML files.
- The `<article>` structure in examples must not change.
- `specs/lys-core.spec.md` must retain all nine token declarations in the table.
- `llms.txt` must remain a valid, concise LLM-readable reference.
- `skill/SKILL.md` must produce working Lys decks when used by an agent.
