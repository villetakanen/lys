# Accessibility — ARIA Roles, Live Region, Focus Management

## Blueprint (Design)

### Context

Accessibility in Lys bridges the gap between "a page of articles" and "an interactive presentation that assistive technology can navigate." Screen readers, switch devices, and other AT need semantic signals — ARIA roles, live region announcements, and predictable focus behavior — to convey the carousel-like interaction model that sighted keyboard users perceive visually.

LLMs generating Lys decks cannot reliably produce correct ARIA attributes, live region markup, or focus management logic. This module makes those decisions structurally, so the generated HTML works for all users without requiring the author to understand WAI-ARIA.

The a11y module has **zero visual output**. It only manipulates ARIA attributes, manages focus, and maintains a live region. It reacts to `lys:slidechange` events emitted by the navigation module — it does not drive navigation itself.

### Architecture

#### ARIA roles and attributes (`src/a11y.ts`)

On initialization, the a11y module sets the following attributes on the container and slides:

**Container (`[data-lys]`):**

| Attribute | Value | Purpose |
|---|---|---|
| `role` | `"group"` | Groups the slides as a logical unit (WAI-ARIA carousel pattern) |
| `aria-roledescription` | `"slide deck"` | Provides a human-readable description of the group's purpose |
| `aria-label` | Derived from context (see below) | Accessible name for the deck |

**`aria-label` resolution order:**
1. If the container already has an explicit `aria-label`, preserve it (author intent).
2. If the container has `aria-labelledby`, preserve it.
3. If the first slide's first heading (`h1`–`h6`) exists, use its text content as the label (e.g., "Quarterly Review" from `<h1>Quarterly Review</h1>`).
4. Fallback: `"Slide deck"`.

**Each slide (`<article>`):**

| Attribute | Value | Purpose |
|---|---|---|
| `role` | `"group"` | Each slide is a group within the carousel |
| `aria-roledescription` | `"slide"` | Screen readers announce "slide" instead of "group" |
| `aria-label` | `"Slide N of M"` | Positional label (e.g., "Slide 3 of 12") |
| `aria-hidden` | `"true"` / removed | Non-current slides are hidden from AT |

**`aria-hidden` behavior:**
- On init, all slides except the current one get `aria-hidden="true"`.
- On `lys:slidechange`, the previous slide gets `aria-hidden="true"` and the new current slide has `aria-hidden` removed.
- This ensures screen readers only encounter the active slide, matching the visual presentation.

**Preserving author attributes:** If an `<article>` already has an explicit `aria-label`, the a11y module does NOT overwrite it. The positional label (`"N of M"`) is only applied when no author-supplied label exists. This allows LLMs or authors to provide meaningful slide names (e.g., `aria-label="Conclusion"`) that take precedence.

#### Live region

The a11y module creates a single live region element for announcements:

```html
<div role="status" aria-live="polite" aria-atomic="true" class="lys-sr-only">
  Slide 3 of 12
</div>
```

- **Injected as a child** of the `[data-lys]` container (not a sibling, to keep it scoped).
- **`aria-live="polite"`** — announcements wait for the screen reader to finish current speech.
- **`aria-atomic="true"`** — the entire region is re-read on change, not just the delta.
- **`class="lys-sr-only"`** — visually hidden but accessible. Styled in `lys.css`.
- **Announcement text:** `"Slide N of M"` where N is 1-based and M is total. If the current slide has an `aria-label` (author-provided), include it: `"Slide 3 of 12: Conclusion"`.

The live region is updated on every `lys:slidechange` event.

#### CSS for visually-hidden live region (`src/lys.css`)

**New CSS class** added to `lys.css`:

```css
.lys-sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
```

This is the standard visually-hidden pattern. It is scoped by class name — not applied globally.

#### Focus management

Focus management ensures keyboard users maintain orientation during slide transitions:

1. **On `lys:slidechange`:** Move focus to the newly active slide.
   - The active slide must be focusable. The a11y module sets `tabindex="-1"` on each `<article>` during init (if not already present). This makes slides programmatically focusable without adding them to the tab order.
   - Focus is moved via `slide.focus({ preventScroll: true })` — `preventScroll` avoids fighting with the navigation module's `scrollIntoView`.

2. **Focus restoration on destroy:** When `destroy()` is called, remove `tabindex` attributes that were added by the a11y module (not author-provided ones). Do not move focus — let the browser handle it.

3. **Focus outline suppression:** Slides fill the entire viewport — a visible focus ring provides no orientation value and creates distracting borders on navigation. The outline is suppressed via `[data-lys] > article:focus { outline: none }`. The `--lys-focus-ring` token remains defined in the public token API for authors who want to apply it in their own CSS. See `specs/focus-ring-bug.spec.md` for rationale.

#### Module boundaries

- **`src/a11y.ts`** — Exports `setupA11y(container, slides)` and returns a handle with `destroy()`. Manages ARIA attributes, live region creation/updates, and focus movement. Listens for `lys:slidechange` on the container.
- **`src/lys.css`** — `.lys-sr-only` class and `article:focus { outline: none }` rule.
- **`src/lys.ts`** — Imports and calls `setupA11y` during construction, calls `handle.destroy()` during `destroy()`.
- **`src/types.ts`** — No changes needed. The a11y module consumes existing `LysSlideChangeDetail`.

#### Integration with other modules

- **navigation** — Navigation emits `lys:slidechange`. The a11y module listens for it. No direct coupling.
- **core** — The a11y module is called from `Lys` constructor and `destroy()`. It receives the container and slides as arguments.
- **transitions** — Fade mode uses the same a11y infrastructure. No changes needed (see `transitions.spec.md`).

### Anti-Patterns

- **Using `aria-live="assertive"`.** Slide changes are routine navigation, not urgent alerts. `"polite"` is correct.
- **Moving focus on every scroll.** The a11y module responds to `lys:slidechange` events, not scroll events. Scroll-snap CSS-only navigation does NOT trigger focus movement (no JS = no a11y module = no live region). This is acceptable per progressive enhancement — CSS-only decks rely on the native `<article>` semantics.
- **Adding `role="presentation"` or `aria-hidden` to the container.** The container is a meaningful group, not decorative.
- **Hiding all slides and showing only the current one via `display: none`.** The layout uses scroll-snap. `aria-hidden` hides from AT; `display: none` would break the layout.
- **Creating multiple live regions.** One per container is sufficient. Multiple regions cause redundant announcements.
- **Setting `tabindex="0"` on slides.** Slides should be focusable programmatically (`tabindex="-1"`) but NOT in the tab order. The container is the tab stop; slides receive focus via JS on navigation.
- **Overwriting author-provided ARIA attributes.** If an `<article>` has an explicit `aria-label`, preserve it. The a11y module only fills in defaults.
- **Announcing on initial load.** The live region should NOT announce "Slide 1 of N" when the deck first initializes — only on subsequent slide changes. Initial state is conveyed by the slide's own ARIA attributes.
- **Using `innerHTML` to set live region text.** Use `textContent` to avoid XSS from author-supplied `aria-label` values.

## Contract (Quality)

### Definition of Done

1. Container has `role="group"` and `aria-roledescription="slide deck"` after init.
2. Container has a computed `aria-label` (author-provided, heading-derived, or fallback).
3. Each slide has `role="group"`, `aria-roledescription="slide"`, and a positional `aria-label`.
4. Author-provided `aria-label` on slides is preserved (not overwritten).
5. Non-current slides have `aria-hidden="true"`; the current slide does not.
6. A live region element exists inside the container with `role="status"`, `aria-live="polite"`, `aria-atomic="true"`.
7. The live region is visually hidden (`.lys-sr-only`).
8. On `lys:slidechange`, the live region text updates to `"Slide N of M"`.
9. On `lys:slidechange`, focus moves to the newly active slide.
10. Slides have `tabindex="-1"` (programmatically focusable but not in tab order).
11. `article:focus` has `outline: none` — no visible focus ring on slides (slides fill the viewport; ring adds no orientation).
12. The `--lys-focus-ring` token is still defined in the token block for author use.
13. `destroy()` removes all ARIA attributes added by the module (not author-provided ones).
14. `destroy()` removes the live region element from the DOM.
15. `destroy()` removes `tabindex` attributes added by the module.
16. The live region does NOT announce on initial load — only on subsequent `lys:slidechange` events.
17. `aria-hidden` updates correctly on every slide change.
18. Zero axe-core critical or serious violations on an initialized deck.

### Regression Guardrails

- A CSS-only deck (no JS) must have no ARIA violations. The `<article>` elements are semantically valid without any Lys-added attributes.
- `destroy()` must revert the container and slides to their pre-init state — no orphaned ARIA attributes, no orphaned DOM nodes.
- An empty deck (0 slides) must not throw during a11y setup. The live region is still created but never updated.
- A single-slide deck must set `aria-label="Slide 1 of 1"` and NOT set `aria-hidden` on the only slide.
- Focus must never be moved without a corresponding `lys:slidechange` event. The a11y module is event-driven, not state-polling.
- The `.lys-sr-only` class must never affect visible layout. It must remain invisible to sighted users.
- The `--lys-focus-ring` token must remain defined in the CSS token block. It is not applied to slides by default but is available for author CSS.

## Known Constraints

- **happy-dom and `:focus-visible`.** happy-dom does not distinguish `:focus-visible` from `:focus`. Focus ring CSS scenarios must be tested in e2e (Playwright) only.
- **Live region timing.** Screen readers may batch or delay live region announcements. E2e tests can verify the DOM text content changes but cannot assert on actual screen reader speech output.
- **`aria-label` vs heading semantics.** Using `aria-label` on the container means screen readers will use it instead of traversing child headings. This is intentional — the deck behaves as a single interactive widget, not a document section.

## Related / Future

- **Presenter mode** (deferred to post-1.0, see `docs/adr/001-defer-presenter-mode.md`) — If implemented, the presenter window would need its own a11y attributes.
- **Fragment / step support** (VISION.md open question) — If incremental reveals are added, the live region announcement format would need to include step information (e.g., "Slide 3 of 12, step 2 of 4").
- **Localization** — The announcement text ("Slide N of M") is English-only. A future enhancement could accept a format function via the API for localized announcements.
